基于Vue的管理后台搭建方案

在这里简单介绍下如何基于vue搭建一个管理后台，并着重介绍一个完整的管理后台需要完善哪些机制以及可拓展的方向。

原料

  • 基本框架：Vue 2.x
  • 数据管理：Vuex
  • 路由管理：Vue-Router
  • 组件库：Muse UI（或者其他支持vue2的组件库）
  • 异步请求库： Axios

工具

  • vue-cli

初始化完的项目的目录是针对vue项目的，这里给出适用于管理后台项目的建议目录如下：

*这里是否引入单元测试以及eslint视具体情况而定

frame/

这个目录存放的是系统外框，具体包括侧边栏、头部、底部等。把这一部分抽出来，以后可以提到框架级别，方便统一修改或者主题切换。

modules/

这个目录存放的是系统模块，以侧边栏的菜单入口为单位。

3.相关机制

接口mock服务

搭建接口mock服务，支持：

  • 本地接口mock
  • 远程接口mock
  • 对接第三方接口管理平台

部署功能

一般部署是由前端同学将项目打包，然后由后端同学部署，但是这种对接的效率并不高。这里要另外先提一下打包机制，目前的打包机制是将文件打包为html文件 + 资源文件，再部署到ftp服务器上。同时每次发布的时候，会有一份名为verison的json文件来记录版本信息，后端同学先获取这份json文件的版本信息，再去获取对应版本的html文件，将其部署到自己的环境，就最终完成了部署。每次前端同学有更新，只需要直接发布到ftp服务器，版本文件同时更新，后端同学就会拿到最新版本的html文件。

权限机制

在管理后台的业务需求种，一般会涉及到权限控制。这里我们根据管理后台的业务场景将权限分为两种：菜单权限和功能点权限。具体需要做三件事：

  • 与后端同学制定好权限接口包括返回的数据接口，是否能清楚地告诉你，哪个模块是不是有权限进入以及进入该模块后有哪些功能点权限
  • 框架获取到权限信息后，将权限信息传递给模块
  • 模块通过权限信息判断哪些区域是否可见或者哪些功能是否具有

共享机制

回想一下，我们一般需要开发好几个后台，是否遇到两个后台需要开发类似的功能的情况？
通常有三种解决方法：

1.代码COPY：可能你会觉得这是速度最快的做法，直接复制之前的组件到新项目中，慢慢的你会发现，随着你项目的增加，代码量在成倍上升，后期极难维护，浪费很多时间。

2.子模块：抽离所有公共组件放入一个子模块（git submodule）中，这种方法虽然解决了重复工作，但对目录结构以及使用者要求较多，不够灵活。

3.通用组件库：基于私有npm库，开发一个内部通用组件库，提供全局引用和按需引用的方式。相比1解决了重复的问题，相比2更加灵活，并且可以通过版本保证稳定性。

4.小结

在各类站点及各类App飞速发展的现在，管理后台的需求无疑也是各种各样，拥有一个稳健并且支持各种拓展机制的管理后台框架很重要。从搭建到开发需求有一段时间了，慢慢积累下来一些东西，希望可以互相激励，也欢迎同学们指导。
