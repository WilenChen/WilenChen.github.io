GitSubmodule子模块使用总结

概念

子模块 = Git仓库

子模块允许你将一个Git仓库作为另一个Git仓库的子目录

它能让你将另一个仓库克隆到自己的项目中，同时还保持提交的独立性

应用场景

问题：解决不同Git项目下的类似甚至相同模块的开发

选项一 Npm：

优点：独立性强（可移植性强）、严格版本控制

缺点：需要独立开发环境、node_modules、打包问题

选项二 Submodule:

优点：*灵活性高、不需要独立开发环境、无需考虑打包问题

缺点：*灵活性高、需要一定的操作规范

常用操作

1.添加子模块

$ git submodule add <remote_url> <local_path>

2.初始化子模块

git submodule init

3.更新子模块

$ git pull update

4.删除子模块

$ git rm -cached <submodule dir>

5.遍历所有子模块

$ git submodule foreach 'git xxx'

注意事项

1.如果要clone一个附带submodule的项目，submodule的文件不会自动随父项目clone出来（其实只会clone出 .gitsubmodles 这个描述文件），还需要执行如下命令取出submodule里的文件：

$ git submodule init

$ git submodule update

或者，一条组合命令（同样适用于嵌套submodule的情况）：

$ git submodule update --init --recursive

2.修改子模块之前，一定要用git checkout master将当前的子模块分支切换都master，然后才能做修改和提交。因为默认情况下，子模块的HEAD是处于游离状态的。
