ReactNative-项目总结

通过 ReactNative 输出 bundle 包集成到现有 iOS 和 Android 应用

技术准备

原生支持

搭建开发环境

借助ReactNative的命令行工具（react-native-cli）

调试工具

  • 开发菜单 - 在真机中可通过摇晃手机调出（远程调试JS、热刷新、元素检查、性能检测）
  • 提示信息 - 应用内的错误与警告提示（红屏和黄屏）
  • react-devtools
  • 在 Android Studio 或者 Xcode 中调试

组件封装

  • NitButton - 按钮组件
  • NitCell - 单元栏组件
  • NitModal - 模态窗组件
  • NitSlider - 滑动组件
  • NitSwitch - 开关组件
  • NitTextInput - 文本输入组件
  • NitDashboard - 仪表盘组件
  • NItColorPicker - 取色盘组件

全局方法封装

  • $apis - 全局api方法
  • $alert - alert框
  • $toast - toast框
  • $confirm - 确认弹框
  • $configs - 全局配置
  • $enums - 全局枚举
  • $utils - 全局函数方法

蓝牙协议解析

基本信息

原生只提供基本蓝牙操作，比如搜索、读写等操作，把设备传递过来的十六进制串解析成对应的可用信息需要JS自己解析，这部分的解析需要根据设备厂商的解析规则进行解析。

固件升级

通过指令组合和分段数据传输来完成一次完整的固件升级。

热更新机制

react-native-update

对于ReactNative输出的bundle包，需要进行对应的版本管理，处理app版本与bundle包版本的对应关系，再由app进行本地bundle包版本检测与比对，从而实现热更新机制。

坑点

  • 【react-native-cli】react-native link 对集成项目无效，只对标准项目生效（react-native init）
  • 【react-native】【Image】Image组件source属性如果是require本地图片，路径不能含变量，网络图片可以含变量拼接
  • 【resact-native】【NativeModules】通过NativeModules调用原生api，原生只能触发一次callback（进度相关的回调，只能使用事件监听的形式）
  • 【Android】【backHandler】官方提供的backHandler，return false的时候，仍需要自己处理返回
  • 【Android】【Text】嵌套使用Text组件的时候，子Text的opacity属性会继承父Text，修改无效
  • 【iOS】【TextInput】0.56.0版本不支持非英文输入（据说0.57.0已修复）
  • 【react-navigation】无法处理右滑手势的拦截，只能监听到路由退出

GitSubmodule子模块使用总结

概念

子模块 = Git仓库

子模块允许你将一个Git仓库作为另一个Git仓库的子目录

它能让你将另一个仓库克隆到自己的项目中，同时还保持提交的独立性

应用场景

问题：解决不同Git项目下的类似甚至相同模块的开发

选项一 Npm：

优点：独立性强（可移植性强）、严格版本控制

缺点：需要独立开发环境、node_modules、打包问题

选项二 Submodule:

优点：*灵活性高、不需要独立开发环境、无需考虑打包问题

缺点：*灵活性高、需要一定的操作规范

常用操作

1.添加子模块

$ git submodule add <remote_url> <local_path>

2.初始化子模块

git submodule init

3.更新子模块

$ git pull update

4.删除子模块

$ git rm -cached <submodule dir>

5.遍历所有子模块

$ git submodule foreach 'git xxx'

(1)切换到子模块目录

(2)进行git操作

(3)提交 –> 新的版本号 (4)外层项目提交 –> 子模块新的版本号

(5)git submodule update

注意事项

1.如果要clone一个附带submodule的项目，submodule的文件不会自动随父项目clone出来（其实只会clone出 .gitsubmodles 这个描述文件），还需要执行如下命令取出submodule里的文件：

$ git submodule init

$ git submodule update

或者，一条组合命令（同样适用于嵌套submodule的情况）：

$ git submodule update --init --recursive

2.修改子模块之前，一定要用git checkout master将当前的子模块分支切换都master，然后才能做修改和提交。因为默认情况下，子模块的HEAD是处于游离状态的。

基于Vue的管理后台搭建方案

在这里简单介绍下如何基于vue搭建一个管理后台，并着重介绍一个完整的管理后台需要完善哪些机制以及可拓展的方向。

原料

  • 基本框架：Vue 2.x
  • 数据管理：Vuex
  • 路由管理：Vue-Router
  • 组件库：Muse UI（或者其他支持vue2的组件库）
  • 异步请求库： Axios

工具

  • vue-cli

步骤

1.初始化三部曲

通过npm安装vue-cli，按以下步骤初始化项目：

2.目录结构

初始化完的项目的目录是针对vue项目的，这里给出适用于管理后台项目的建议目录如下：

*这里是否引入单元测试以及eslint视具体情况而定

frame/

这个目录存放的是系统外框，具体包括侧边栏、头部、底部等。把这一部分抽出来，以后可以提到框架级别，方便统一修改或者主题切换。

modules/

这个目录存放的是系统模块，以侧边栏的菜单入口为单位。

3.相关机制

接口mock服务

搭建接口mock服务，支持：

  • 本地接口mock
  • 远程接口mock
  • 对接第三方接口管理平台

部署功能

一般部署是由前端同学将项目打包，然后由后端同学部署，但是这种对接的效率并不高。这里要另外先提一下打包机制，目前的打包机制是将文件打包为html文件 + 资源文件，再部署到ftp服务器上。同时每次发布的时候，会有一份名为verison的json文件来记录版本信息，后端同学先获取这份json文件的版本信息，再去获取对应版本的html文件，将其部署到自己的环境，就最终完成了部署。每次前端同学有更新，只需要直接发布到ftp服务器，版本文件同时更新，后端同学就会拿到最新版本的html文件。

权限机制

在管理后台的业务需求种，一般会涉及到权限控制。这里我们根据管理后台的业务场景将权限分为两种：菜单权限和功能点权限。具体需要做三件事：

  • 与后端同学制定好权限接口包括返回的数据接口，是否能清楚地告诉你，哪个模块是不是有权限进入以及进入该模块后有哪些功能点权限
  • 框架获取到权限信息后，将权限信息传递给模块
  • 模块通过权限信息判断哪些区域是否可见或者哪些功能是否具有

共享机制

回想一下，我们一般需要开发好几个后台，是否遇到两个后台需要开发类似的功能的情况？
通常有三种解决方法：

1.代码COPY：可能你会觉得这是速度最快的做法，直接复制之前的组件到新项目中，慢慢的你会发现，随着你项目的增加，代码量在成倍上升，后期极难维护，浪费很多时间。

2.子模块：抽离所有公共组件放入一个子模块（git submodule）中，这种方法虽然解决了重复工作，但对目录结构以及使用者要求较多，不够灵活。

3.通用组件库：基于私有npm库，开发一个内部通用组件库，提供全局引用和按需引用的方式。相比1解决了重复的问题，相比2更加灵活，并且可以通过版本保证稳定性。

一系列的自动化命令

也就是cli，之前已经用过vue-cli了，所以应该很清楚是做什么的。在管理后台的需求量越来越多的情况下，敏捷开发很有必要。具体点，初始化菜单模块的时候，你应该还是从零开始写代码吧，好吧，你说复制代码…如果只是敲一行命令，我就帮你把开发菜单模块需要的文件都创建好，并且已经写好demo代码，你喜欢么？将一些重复的事情命令化变得高效，比如一系列的自动化命令。

4.小结

在各类站点及各类App飞速发展的现在，管理后台的需求无疑也是各种各样，拥有一个稳健并且支持各种拓展机制的管理后台框架很重要。从搭建到开发需求有一段时间了，慢慢积累下来一些东西，希望可以互相激励，也欢迎同学们指导。
